Mini SOC Automation Platform

This project started as a simple Python authentication log analyser and gradually evolved into a beginner SOC-style automation platform designed to simulate real cybersecurity investigation workflows.

The system analyses authentication log files, detects suspicious login behaviour, stores investigation data inside a SQL database, collects Windows security evidence using PowerShell, correlates security alerts into incidents, and automatically generates SOC-style incident reports.

The main goal of the project was to better understand how modern Security Operations Centres (SOCs) investigate threats, analyse logs, detect attacks, and manage security incidents using automation.

---
 Authentication Log Analysis

The first stage of the project focused on analysing authentication logs using Python.

Authentication logs contain records of successful and failed login attempts generated by websites, applications, or systems. The Python script reads these logs, processes every line individually, extracts important information such as usernames, IP addresses, login status, dates, and timestamps, and then analyses the events for suspicious behaviour.

The system specifically focuses on detecting brute force attacks by identifying repeated failed login attempts coming from the same IP address.

The project also tracks successful logins, allowing the system to compare failed and successful authentication behaviour during investigations.

---

Brute Force Detection

One of the main cybersecurity goals of the project is detecting brute force attacks.

The system monitors repeated failed login attempts and uses threshold-based detection logic to classify suspicious behaviour into different severity levels such as HIGH and CRITICAL.

For example, repeated failed logins from a single IP address may indicate that an attacker is attempting to guess passwords through automated login attempts.

The platform also detects situations where multiple failed login attempts are later followed by a successful login, which may indicate a possible account compromise or successful brute force attack.

---

SQL Investigation Database

The second stage of the project introduced SQL databases into the architecture.

Originally, the system only stored information temporarily inside Python variables while the script was running. Once the script stopped, most of the investigation data disappeared.

To solve this problem, SQLite was integrated into the platform to provide long-term structured storage.

The database stores authentication events, investigation findings, and generated alerts inside structured tables using rows and columns. This allows the platform to preserve historical evidence and perform investigation queries later.

The SQL integration allows the system to identify:

* top attacking IP addresses,
* most targeted users,
* repeated attack patterns,
* and suspicious authentication behaviour over time.

This stage introduced the concept of investigation-focused cybersecurity workflows instead of simple temporary log analysis.

---

 Windows Security Evidence Collection

The third stage of the project introduced PowerShell-based Windows security evidence collection.

Using PowerShell, the system gathers information directly from the Windows operating system, including:

* computer name,
* Windows Defender status,
* local administrator accounts,
* and running processes.

This information is exported into JSON format and later analysed by Python.

The purpose of this stage was to simulate endpoint visibility and system awareness, which are important parts of real SOC investigations.

Rather than analysing only login logs, the platform now also collects system-level evidence that may help analysts understand whether a machine has been compromised or misconfigured.

---

SIEM-Style Correlation

The fourth stage of the project introduced basic SIEM-style event correlation.

Instead of treating alerts as isolated events, the platform now attempts to connect related suspicious activities together to build incidents.

For example, the system can correlate:

* repeated failed logins,
* successful authentication after failures,
* targeted administrative accounts,
* and suspicious activity patterns.

This allows the platform to move beyond simple event detection into more realistic incident-based investigation logic.

The project also introduced risk scoring, where different alerts contribute to an overall incident risk level depending on their severity.

---

 Automated SOC Reporting

The final stage of the project introduced automated SOC-style reporting.

The platform automatically generates investigation reports containing:

* detected alerts,
* incident summaries,
* correlated attack information,
* severity levels,
* risk scores,
* usernames,
* IP addresses,
* and investigation descriptions.

These reports simulate the type of documentation and investigation summaries analysts may create during real cybersecurity incident response workflows.
